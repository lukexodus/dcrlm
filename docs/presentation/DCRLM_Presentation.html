The Problem02 / 17
Why this is hard

Distributed systems break the rules we rely on

No shared clock

Each machine has its own hardware clock. Clocks drift apart. You cannot use wall-clock time to order events across machines.

No shared memory

Machines cannot read each other's variables. All coordination must happen through explicit messages over a network.

Partial failures

One machine can crash while others keep running. The system must detect and recover from this gracefully.

The core challenge: Only one worker may use the shared resource at any moment — but there is no mutex, no semaphore, and no shared state. It must be built entirely from message passing.

Architecture03 / 17
Three-process model

Each process has exactly one job

Naming Server

Runs first
naming_server.py

Maps logical service names to physical ip:port addresses. Handles REGISTER and LOOKUP. Not involved after startup.

Only hardcoded port: 5000

Lock Manager

Central engine
lock_server.py

Maintains the priority queue. Grants and releases the lock. Broadcasts queue updates. One thread per worker, all state under a mutex.

Watchdog: auto-release after 30s

Worker Client

N instances
worker_client.py

Interactive terminal. Resolves the Lock Manager via naming, connects, sends requests with Lamport timestamps. Two threads: input + listener.

Shared lib: utils.py

① First

→
② Second
Lock Manager
lock_server.py

Registers its dynamic port with the naming server. Workers can now resolve where to connect.

→
③ Any order
Workers × N
worker_client.py

Each worker looks up the Lock Manager address at startup. Multiple workers can join in any order.

Why it matters: If a Worker starts before the Lock Manager registers, the LOOKUP will fail. If the Lock Manager starts before the Naming Server, registration will fail. Startup order is a hard dependency, not just a convention.

Integration05 / 17
How it all fits together

End-to-end message flow

1
Naming Server starts
Binds to port 5000. Holds an empty registry. Waits for REGISTER and LOOKUP requests.
2
Lock Manager registers
REGISTER lock.server.main ip:port → NS stores the address. Lock Manager now listens for Workers.
3
Workers resolve and connect
LOOKUP lock.server.main → NS returns address → Worker opens TCP connection → sends hello.
4
Worker requests lock
Worker increments Lamport clock, sends request_lock {timestamp, worker_id}. LM applies Rule 3, enqueues request, sorts queue.
5
LM grants lock & broadcasts
Sends lock_granted to queue head. Sends queue_update to all others so they know their position.
6
Worker releases lock
Sends release_lock. LM removes from queue, grants next in line. Cycle repeats.

Naming07 / 17
Concept 1 — Location independence

Naming Server: decoupling identity from address

Hardcoding 192.168.1.45:9000 into every worker is brittle. If the Lock Manager restarts on a different port, every client breaks. The naming server decouples the logical name from the physical location.

# Registry contents
registry = {
  "lock.server.main": ("192.168.1.45", 9000)
}
The naming server's port (5000) is the only hardcoded value in the entire system. Everything else is discovered at runtime.
Protocol
Lock Manageron startup
REGISTER lock.server.main
         192.168.1.45 9000
Workeron startup
LOOKUP lock.server.main
→ 192.168.1.45 9000

After lookup, the worker connects directly to the Lock Manager. The naming server is not contacted again during the session.

Communication08 / 17
Concept 2 — Message-oriented communication

JSON over TCP with length framing

Message examples
// Worker → Lock Manager
{ "type": "request_lock",
  "worker_id": "WA",
  "timestamp": 7, "resource": "gpu_0" }
// Lock Manager → Worker (broadcast)
{ "type": "queue_update",
  "queue": [[ "WA",7],["WB",9]],
  "your_position": 2 }
TCP framing problem

TCP is a byte stream, not a message protocol. recv(4096) may return half a message or two merged together.

Length-prefix scheme
4 bytes
JSON payload (N bytes)
recv_json reads 4 bytes → unpacks length → reads exactly N bytes. Atomic, regardless of TCP fragmentation.
Async by design

Workers use two threads — one for user input, one listening for server broadcasts. Sender and receiver are decoupled in time.

Lamport Clocks09 / 17
Concept 3 — Logical time

Three rules that order events without a shared clock

LamportClock in utils.py
1
Local event
clock = clock + 1
2
Send message
clock++; attach to msg
3
Receive message (key rule)
clock = max(clock, T) + 1
Tie-breaking: When timestamps match, lower worker_id wins. "WA" < "WB" < "WC" — deterministic and fair.
Queue in action — WA and WB tie at T=3
LM t=0Initial state. Queue empty.
recv WBT=3, WB → clock = max(0,3)+1 = 4
recv WAT=3, WA → clock = max(4,3)+1 = 5
recv WCT=5, WC → clock = max(5,5)+1 = 6
Sorted queue → (ts, id)
WA
(3, WA) ✓
WB
(3, WB)
WC
(5, WC)

Physical arrival order is irrelevant. WA arrived second but wins because "WA" < "WB". Logical order is what matters.

Summary14 / 17
What we built

Three concepts. One working system.

DNS
Naming Server

Logical names decouple identity from address. Workers never hardcode locations. Re-registration on crash.

TCP
Message Framing

Length-prefix framing on all messages. JSON for readability. Async two-thread workers for non-blocking I/O.

T+1
Lamport Clocks

Causal order without a shared clock. Priority queue + tie-breaking by ID. Fair, deterministic, no starvation.

Key property: exactly one worker holds the lock at any moment, regardless of network timing, physical arrival order, or clock drift — enforced entirely through message passing.
